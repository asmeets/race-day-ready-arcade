Use this page to frame what success looks like for a session.

## Learning Goals

By the end of the experience, learners should be able to:

- describe how a code change affects what happens in the game
- explain at least one engineering tradeoff involving speed, control, efficiency, or sustainability
- test a change, observe the result, and revise their design
- connect gameplay choices to real careers in software, testing, strategy, sustainability, or data

## Facilitation Goals

Adults running the session should aim to:

- keep the experience moving with short instructions and fast test loops
- point learners to a small number of visible variables, functions, or blocks
- normalize debugging as part of the process
- use discussion to connect game performance with engineering reasoning

## Design Principles Behind the Project

- **Accessibility first:** short text, clear feedback, simple controls, and recoverable failure states
- **Remixability:** visible tuning values and readable code patterns learners can trace
- **Event reliability:** predictable behavior and low-friction facilitation
- **Meaningful tradeoffs:** gameplay balances performance, handling, and sustainability together

## Evidence of Success

You are on track if learners can say what changed, why they changed it, and what happened next.
